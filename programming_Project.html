Localization Algorithm (Java)

This project started out of necessity for my FTC team's 2021-22 season robot. I developed the entire system and programming, but couldn't quite get the system to work properly due to time constraints. I plan to continue working on this algorithm in conjunction with other, personal, robotics projects in the future.

The Problem

On the 2021-22 FTC season field, there were a set of small pipes that robots could either navigate around or over. To go around, robots would need to be at least two inches smaller than the maximum size limits. Our team elected to go over the bars, which means we would need a different method than the normal FTC localization method. Most robots with dedicated localization systems use dead-wheel odometry, where a set of un-powered wheels are tracked with dedicated encoders. However, because we were going over the bars, our odometry would have a high amount of error as soon as the dead-wheels leave the ground. So, we decided to use a new system that wouldn't rely on contact with the ground.

The Solution

We developed a system that uses a series of seven REV distance sensors to detect the field walls. The four sensors on the corners of the robot were mounted on servos that would rotate them to face forward and backward. This gave us three sensing points on three of the four sides on our robot. By using several sensors on one side, we would be able to both align the robot to the field as desired and find our position based on two of the four field walls (because the sensors can't see the full distance of the field).

The Code

General Function

The system works by turning the servos and taking samples of the distances from each sensor (two sides at a time) in half-second intervals. The data from the sensors is used to determine the robot's current position on the field, which is based off a coordinate system with the origin in a specific corner. Each of the four starting positions has a preferred set of measurements to use, if they exist, to minimize interference from game elements or other robots. Other autonomous instructions are implemented through Java methods that move the robot in straight lines and rotate it, as needed.

Discrepancy Handling

The biggest challenge with this system is dealing with sensor measurements that don't match (within a margin of error). As the robot moves a PID loop corrects any drifting to either side, but when a sensor picks up another robot or a field element, the program needs to deal with a larger discrepancy. The algorithm determines whether the discrepancy is large enough that there's something in the way and tracks how long it has been in the way to determine whether it is a robot or field element (i.e. is it stationary or not). If it is stationary the robot avoids it, but other robots will be waited out for a brief period.

Issues

The current position recognition system works well if there are no other elements in the field, but that's never the case. In matches, the algorithm would get confused when determining what things to avoid and delay the autonomous too long. The sensors on servos would occasionaly take a reading too early or late, so it was measured while not on a specific side, which caused discrepancy handling to fail.
